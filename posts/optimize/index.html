性能优化

官网有一个 性能专题。有一系列视频。youtube上讲的比较乱。udacity上同一个人讲的比较有条理，还有相应的代码举例。当然文本的方式读起来更快,更符合自己的节奏。

1. 绘制

CPU处理input,measure,layout等之后，最后都要通过GPU栅格化绘制每一个像素点。所以就可以分别优化CPU，GPU：减少measure,layout的时间（ 更加扁平的布局）,在必要的时候才调用invalidate()；不要过度绘制（减少重复绘制的背景，限制绘制区域clipRect）。

绘制调试，开发者选项中有2个开关，一个用来看绘制各个阶段的时间，另一个用来查看过度绘制。

2. 计算

  • 比较耗时的计算可以使用缓存，批处理。 这里更接近数据结构与算法，用空间复杂度，换取时间复杂度。比较典型的例子就是哈希表的来降低时间复杂度。

  • UI线程阻塞，另外创建线程解决来解决。

3. 内存

  • 不能自动GC引起的内存泄露。用不到的对象要置null。

  • 频繁GC引起的内存抖动。初始分配，之后复用。

操作系统层面

操作系统层面，计算机的资源有划分为：CPU（计算），内存（存储），其他（I/O设备,总线等）。

对于CPU而言，分为单核和多核。可以引出线程的2个概念。

  1. 并发（concurrency）： 交替执行 对于单核CPU而言。比如：2个线程交替获取CPU的时间片。
  2. 并行（parallelism）： 可以一起运行 对于多核CPU而言。比如：线程1获取CPU1的时间片，线程2获取CPU2的时间片在同一时间共同执行。

对于内存而言，分为RAM和ROM。

  1. RAM：运行内存。其中内存的分配，映射，回收都是需要很深入研究的话题。Android中有2个常见的问题：内存的泄露和内存的抖动。
  2. ROM：磁盘。用于文件，数据库的存放。

进程是资源分配的基本单位.线程是执行和调度的基本单位。这句话翻译翻译就是：CPU，内存，其他资源都是以线程为基本单位进行分配的，而线程是使用CPU的基本单位。 所以理论上，在线程的优先级相同的情况下，进程中的线程越多，能获取CPU执行的时间越长。但线程越多，程序的同步逻辑更复杂，创建和销毁线程的时间消耗也就越多。

线程根据相关资源（主要是CPU和I/O设备,总线，互斥资源。为什么没有内存呢？ 如果没分配内存，怎么会有线程）的有无，划分为：

  • 就绪： 没有CPU，其他资源都有。
  • 运行： 获取CPU ，其他资源都有。
  • 阻塞： 没有CPU，没有其他资源。

可以看到：只有当线程的其他资源都满足了，才会分配CPU，从这个角度也一定程度的说明了：线程是执行和调度的基本单位。

Java层面

Java虚拟机中的线程是基于优先级的抢占式调度的，但到了操作系统层面却不一定和Java虚拟机表现一致，操作系统可能是基于时间片轮转的非抢占式调度。所以Java的调度算法是不能保证的。

Java在语言层面提供了Thread类用于创建线程，有且仅有Thread类能创建线程。还提供了一些用于定义线程处理逻辑和更好的管理线程的类和接口：Callable，Runnable，Future，Executor。

Java为了方便还扩展了线程的状态：

// Java中线程的状态 多了：新建，终止，TIMED_WAITING
public enum State {
	// The thread has been created, but has never been started.
	NEW, 新建
	// The thread may be run.
	RUNNABLE,
	// The thread is blocked and waiting for a lock.
	BLOCKED,
	// The thread is waiting.
	WAITING,
	// The thread is waiting for a specified amount of time.
	TIMED_WAITING,
	// The thread has been terminated.
	TERMINATED  终止
}

Thread 中的stop()，suspend()，resume()，由早期Java的考虑不周，会引起不安全（没有解决同步和死锁）。已弃用。

Java中“锁”表示程序中互斥对象和类，代码块的访问权限。例如：获得某个对象的“锁” == 可以读取该对象（写入则要看锁的类型）

sleep()和join()都是通过synchronized关键字实现，本质就是访问互斥对象。例如：

// 比如： 主线程中调用另一个线程的join()方法，lock对象的wait()方法，使主线程进入阻塞状态。
public final void join() throws InterruptedException {
	synchronized (lock) {
		while (isAlive()) {
			lock.wait();
		}
	}
}

yield()：线程（相对于优先级相同的线程）放弃运行，将CPU的控制权让出。

Thread 没有直接控制线程状态的方法（stop()等不安全）。都是通过控制互斥资源，间接控制线程的状态（sleep()和join()）。如果一个线程要停止，只能使用中断机制，在run()方法里面写判断逻辑，跳出run()方法,使其正常退出。其实也不优雅。

Thread的start()方法会调用run()方法。run()方法默认会调用Runnable接口。

Executor线程池是在1.5之后引入，目的就是更好的控制线程，比如优雅的退出线程，复用线程(不用在创建和销毁线程池上花费时间)等。 Callable，Future本质上都是在Thread里的run()方法里做一些事，和中断处理类似。

Java中线程的安全

  • 线程角度：

线程安全要保证3个特性： 原子性，可见性(读结果一致)，有序性（禁止指令重排，在一个线程内按顺序执行）。 两个原子性的操作组合在一起不一定具有原子性。例如：

// 不为空 加入map 加入操作 containKey和put是原子操作 需要加锁才能保证安全
 if (!map.containsKey(key)){
	 map.put(key, value); // containKey和put是原子操作 需要加锁(synchronized)才能保证安全
 }
  • 资源角度：

synchronized关键字锁的是相应的资源（某一个对象，静态方法，类对象），没有获取相应资源的线程会被阻塞。

有一个比较绕的：对于同一个类A，线程1争夺A对象实例的对象锁，线程2争夺类A的类锁(A的class对象)，这两者不存在竞争关系。相互不干扰。

静态方法和类锁表现一致。

Java1.5以后引入java.util.concurrent并发包。在读和写方面做了优化，允许多个一起读 ;一个写和多个读一起。 提高了并发性 ，减小了锁粒度。 （没有仔细研究，待后续研究）
